**Deviations from the phase plan:**
1. **Login UX is name-only, not "name (+ staff_code where names collide)."** Discussed directly with the user: `00_ARCHITECTURE.md` §5 describes a name picker disambiguated by staff_code (e.g. "John (04)"). The user judged real name collisions unlikely enough in this 5-person roster to skip surfacing staff_code in the UI at all. **Schema is unchanged** — `staff_code` still exists, still builds the synthetic email, per the user's explicit choice to keep it internal rather than remove it. The login route resolves a picked name to a `staff_code` server-side (picking the lowest `staff_code` if a genuine collision ever occurs — an accepted V1 tradeoff, not handled with a disambiguation UI). If real name collisions become a problem, resurrecting the staff_code-suffix UI is a small, contained change to `app/login/page.tsx` and the login route — no schema/migration work needed.
2. **Added `public.login_roster()` — a function not in `01_DATA_MODEL.md`.** The login screen needs a name picker, but `users` RLS (`users_select_own_or_admin`) requires `auth.uid()` — an anonymous, pre-login client can read nothing. Flagged to the user before building; agreed approach was a narrow `security definer` function (same pattern as `canteen_supplied_total()`) returning **only `name`**, granted to `anon`. No staff_code, PIN, role, or location is ever exposed pre-login. Documented in the migration's own comment block; **`01_DATA_MODEL.md` itself was not amended with a new numbered section for this** (unlike the grants gap below) since it's small and self-contained — flagging here is the record.
3. **Added `supabase/migrations/20260710110004_grants.sql` — a real, load-bearing gap in the original schema doc.** `01_DATA_MODEL.md` documents RLS policies in full but never states the baseline `GRANT` statements every Supabase project needs alongside them. RLS and table-level privileges are independent Postgres mechanisms: `service_role` has `rolbypassrls = true` (confirmed via `pg_roles`) but that does **not** imply table-level grants. A dashboard-created Supabase project gets these invisibly; a project built by hand-writing migrations from a doc (this one) does not. Discovered when `scripts/seed-staff.ts`'s service-role insert into `public.users` failed with `permission denied for table users` despite correct RLS and RLS-bypass. Fixed with `grant all on all tables/sequences/routines in schema public to anon, authenticated, service_role` plus matching `alter default privileges` so future migrations' new objects inherit the same grants automatically. **`01_DATA_MODEL.md` §4 was amended** with a new "§4.1 Schema-level grants" subsection documenting this, per CLAUDE.md's mid-project change-handling rule — this is not optional context, any future phase adding a table needs to know grants aren't automatic here.
4. **`middleware.ts` triggers a Next.js 16 deprecation warning** ("The middleware file convention is deprecated. Please use proxy instead") at build time. This is a rename in Next.js 16, not a removal — the build succeeds, only a warning is printed. Kept `middleware.ts` as-is because `CLAUDE.md`'s Project Structure section explicitly names this file `middleware.ts`; renaming to `proxy.ts` unilaterally would itself be an undocumented deviation. Flagging for a human decision: rename in a future phase (and update `CLAUDE.md`) once it's confirmed `proxy.ts` is stable/recommended, or leave as-is until Next.js actually removes the old convention.
5. **`app/page.tsx`'s originally-scaffolded content (Phase 1's placeholder `<Wordmark />` render) was replaced**, per this phase's explicit scope ("app/page.tsx redirects to /login or /entry based on auth state").

**Challenges faced:**
- **Docker/WSL environment setup took real wall-clock time.** Docker Desktop's WSL integration wasn't initially enabled for this distro ("Ubuntu"), and even after enabling it, the local Supabase stack's first-time image pull (13 separate images: Postgres, GoTrue, PostgREST, Realtime, Storage, Kong, Studio, imgproxy, edge-runtime, logflare, vector, postgres-meta, mailpit) took a long time on this connection and had to be retried once after Docker itself was restarted mid-pull (another running project's containers were also competing for resources until the user freed them). Not a code problem, just genuinely slow — resolved by waiting it out with periodic checks rather than assuming something was broken.
- **The grants gap (Deviation 3) was the most consequential finding this phase.** It would have silently blocked every future phase's server-side writes (service-role inserts, admin operations) the first time anyone tried them against a freshly-provisioned Supabase project built from this doc's migrations alone — not just staff seeding. Caught early specifically because Phase 2 seeds real data as an acceptance criterion, rather than only checking that migrations *apply* without SQL errors (they did apply cleanly; the failure was at the privilege layer, invisible until an actual write was attempted as a non-superuser role).
- **Supabase-js's typed query builder loses column-level type information across certain chains** (`.select().eq().order().limit()`, `.select().eq().single()`) under TypeScript strict mode, collapsing to `never`. Confirmed this is specific to the builder chain shape (an isolated single-line query typechecks fine) rather than a real schema mismatch, by testing a minimal reproduction. Fixed with explicit `Awaited<typeof query>` type annotations at the point of destructuring, which is more robust than a hand-written shape annotation since it still surfaces the query's real error type.
- **Login UX for name collisions was discussed and simplified mid-session** — see Deviation 1. The user made a direct judgment call (5-person roster, collision unlikely) rather than building the full staff_code-disambiguation UI `00_ARCHITECTURE.md` describes. Handled by asking rather than assuming, per the instruction to raise decisions genuinely belonging to the user.
