# Phase 5 — Canteen Weekly Entry & Expenses — 2026-07-11

**Status:** Complete

**What was built:**
- **`supabase/migrations/20260711140001_canteen_entry_function.sql`** — `public.save_canteen_stock_entry()`, the canteen-side sibling of Phase 4's `save_stock_entry()`. Same atomic upsert-plus-oversell-recheck shape, but: (a) opening_stock carries forward from the prior *week's* closing_stock (`entry_date < p_week_start`), not the prior day's; (b) `added_stock` for a `canteen_supplied` item is never trusted from the client — the function calls `public.canteen_supplied_total()` itself and uses that; for `canteen_independent` items it uses the caller-supplied value; (c) no `sent_out` parameter (canteen never forwards stock onward — stays 0 via the column default). `security invoker`, same rationale as Phase 4's functions.
- **`lib/calculations.ts`** — added `weekStartMonday(date)` and `weekEndSunday(weekStartISO)`, pure functions implementing the "Monday of the current ISO week" convention. All date math done in UTC explicitly (`getUTCFullYear`/`getUTCMonth`/`getUTCDate`, not the local-timezone getters) — a real bug was caught and fixed here during testing, see "Challenges faced." 8 new tests in `lib/calculations.test.ts`, including an explicit month-boundary case.
- **`lib/validation.ts`** — `canteenStockEntryLineSchema`/`canteenStockEntriesSaveSchema` (no `sent_out` field, unlike the restaurant schema), `expenseCategorySchema`, `expenseSchema`.
- **`app/api/stock-entries/route.ts`** — extended, not replaced. `GET` now branches on `user.location`: canteen requests normalize the client's `date` param to that week's Monday server-side, and additionally return `week_end` + a `supplied_totals` map (one `canteen_supplied_total()` RPC call per `canteen_supplied` item in the response) so the screen can render the read-only aggregate before any save has happened. `POST` branches to a new `saveCanteenEntries()` helper for canteen callers, which re-derives `entry_date` from `weekStartMonday()` server-side (never trusts the client's date verbatim — this mattered, see "Challenges faced") and calls `save_canteen_stock_entry()` per line, passing `p_is_canteen_supplied` from a fresh `items.supply_type` lookup so the function knows which items to aggregate vs. accept as typed.
- **`app/(staff)/entry/CanteenEntryClient.tsx`** — a genuinely separate component from `EntryClient`, not a `cadence` prop variant (per Phase 4's explicit instruction to weigh this rather than force-fit). Reuses `entry.module.css`, `Stepper`, `TillStrip`, `SearchBar`, `Input`, `Toast`, `EmptyState` — no new base components. Renders the "Weekly reconciliation" pattern header (overline + "Week of Jul 6–12" range label), one row per item with: opening (read-only), `Added stock` (read-only pill for `canteen_supplied`, a normal `Stepper` for `canteen_independent`), `Quantity sold` (`Stepper`), and the same collapsed wastage pattern as Phase 4. No category chips — canteen's item list is short enough (11 items in seed data) that the daily screen's category filter wasn't worth porting; search alone was judged sufficient (a design decision, not an oversight — flagged here per the design-system-gap convention, though it's a simplification, not a gap).
- **`app/(staff)/entry/page.tsx`** — now branches on `user.location`: canteen renders `CanteenEntryClient`, restaurant renders the existing `EntryClient`. Same route (`/entry`), per `docs/SCREENS.md`.
- **`app/(staff)/entry/entry.module.css`** — added `.weeklyHeader`/`.weeklyOverline`/`.weekRangeLabel`/`.readOnlyValue`. Initially built with `color-surface-sunken` per a literal reading of the pattern spec, but `StaffShell`'s own page background is *already* `color-surface-sunken` — a real rendered screenshot (see "Challenges faced") showed the header band was visually indistinguishable from the page. Fixed to `color-surface-raised` (white) + a 1px border, which now reads as a clearly distinct band against the sunken page, confirmed via a second screenshot.
- **`app/api/expenses/route.ts`** (new) — `GET ?date=` (caller's own location + date, most recent first) and `POST` (single expense: category/amount/note; `location`/`created_by`/`expense_date` always server-derived, never accepted from the client body — there is no field for a crafted cross-location request to spoof, verified live, see "Gating checklist results").
- **`app/(staff)/expenses/{page.tsx,ExpensesClient.tsx,expenses.module.css}`** (new) — single-submit-plus-running-list pattern (confirmed with the user before building, see the plan-back conversation): `CategoryChips` for the four `expense_category` values, an amount `Input`, an optional note `Input`, a primary "Add expense" button. Below the form, today's already-logged expenses for the caller's location, with a running total. This diverges from Phase 4's till-strip batch-save pattern deliberately — expenses have no natural fixed set of "rows" to pre-populate and occur sporadically, not as a daily sheet.
- **RLS fixes** (see "Challenges faced" for how each was found) — two new migrations:
  - **`20260711150001_canteen_current_week_update_policy.sql`** — the existing same-day-owner UPDATE policy (`entry_date = current_date`) only covered the *daily* cadence. Replaced with a policy that also accepts `location = 'canteen' and entry_date = date_trunc('week', current_date)::date`, so canteen staff can correct a stepper tap anywhere within the current week, not just on the literal calendar day matching `entry_date`.
  - **`20260711160001_insert_current_period_only.sql`** — a genuinely pre-existing gap, not introduced this phase: `stock_insert_scoped` and `ingredient_entries_insert_restaurant` had **no date restriction at all**, only the UPDATE-path policies did. Since `save_stock_entry()`/`save_canteen_stock_entry()` upsert, a *first-ever* save for a given item+location+date goes through the INSERT branch — meaning a client could write an entry for an arbitrary past or future date/week (verified live: a manufactured future-Monday POST succeeded with `200` before this fix). Tightened both INSERT policies to require the same "current period" check the UPDATE policies already enforce.

**UI-expert lens adopted:** for `/entry` (canteen), a **forms/reconciliation specialist** lens, not Phase 4's POS/retail lens — explicitly stated in the plan-back to the user before building. Read-only aggregate fields are styled to look deliberately non-interactive (bordered, inset), matching "this is a periodic check, not a rapid-tap flow." For `/expenses`, a **forms/CRUD specialist** lens — the primary action is a single considered entry, not a fast sweep through many rows, so the screen is a form-plus-list rather than a sheet.

**Deviations from the phase plan:**
1. **`CanteenEntryClient` is a separate component**, not a `cadence` prop on `EntryClient` — an explicit decision Phase 4's context file asked this phase to make deliberately rather than defaulting either way. Reasoning: the two screens differ in almost every row-level field (no `sent_out`, per-item read-only-vs-editable `added_stock` branching, no store-manager variant, no category chips), so a shared component would need as much conditional logic as two components have separate logic — not a net simplification.
2. **Expenses use single-submit-plus-list, not a batch till-strip save** — confirmed with the user via `AskUserQuestion` before building (see conversation). `04_PHASE_PLAN.md` doesn't specify a save pattern for expenses; this was flagged as a genuine ambiguity, not silently decided.
3. **Two new RLS migrations beyond what the phase plan anticipated** (`20260711150001`, `20260711160001`). The first is a direct, necessary consequence of canteen's weekly cadence existing at all (Phase 4's same-day-owner fix was daily-only by construction). The second is a real pre-existing gap in Phase 4's schema that had gone unexercised because Phase 4's UI only ever sent "today" as the date — Phase 5's canteen flow was the first code path to ever send a client-influenced date to the INSERT path, which is what surfaced it. Both found via live testing per CLAUDE.md's testing discipline, not by inspection.
4. **No category chips on the canteen screen** (see "What was built" above) — a simplification given the shorter item list, not built into the design system spec either way. Not flagging as a hard design-system gap since `CategoryChips` remains fully available if a future phase's data set grows enough to need it.

**Challenges faced:**
- **Docker/Supabase local stack wasn't running at the start of this session** (WSL's Docker integration wasn't enabled for this distro). Flagged explicitly to the user rather than silently skipping live-testing or faking results; the user enabled it mid-session and testing proceeded in full — see the conversation. Static checks (lint/typecheck/test/build) were run and passed before this was resolved, so code quality wasn't blocked on it, but the RLS/live-DB gating item genuinely could not have been honestly marked "pass" without it.
- **`weekStartMonday` had a real timezone bug**, caught by its own unit test, not by inspection: using `date.getFullYear()/getMonth()/getDate()` (local-timezone getters) on a `Date` constructed from a UTC ISO string silently shifted the computed Monday by a day depending on the server's local timezone — e.g. `2026-07-12T23:00:00Z` (a Sunday in UTC) resolved to the wrong week because the local reconstruction saw it as already the 13th. Fixed by using the UTC getters (`getUTCFullYear()` etc.) throughout. Worth remembering: any date-only logic in this codebase must reconstruct via UTC getters, not local ones, given entries are stored as bare `date` columns with no timezone component.
- **The INSERT-policy gap** (see "Deviations" #3 and gating checklist item (j)) — this was the same class of bug as Phase 4's "Post-review round" same-day UPDATE gap (a policy correct for the restaurant's daily assumption silently didn't generalize), but on the INSERT side, and pre-existing rather than newly introduced. It was only found because Phase 5's canteen `POST` was the first write path to ever pass a client-influenced date through to a *first* save for a row (Phase 4's UI hardcodes "today," so its INSERT path never got a chance to be tested with a non-today date). This is the exact "obvious but untested scenario" CLAUDE.md's brief asked to think through in advance — in this case it was found by literally trying a future-week POST as part of the planned test matrix, not by accident.
- **The weekly-header sunken-band visual bug** — caught only because of the "verify, don't guess" rule: the CSS looked correct against the spec on paper (`color-surface-sunken` background, matching the pattern doc's literal words), but a real screenshot showed it was indistinguishable from `StaffShell`'s own sunken page background. Fixed and re-verified with a second screenshot before reporting done, per CLAUDE.md's explicit instruction from Phase 4's incident.
