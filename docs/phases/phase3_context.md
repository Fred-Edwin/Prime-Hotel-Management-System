# Phase 3 — Admin Management Shell — 2026-07-10

**Status:** Complete

**What was built:**
- **Admin shell/navigation** — `app/(admin)/layout.tsx` (server component, re-checks `role === 'admin'` server-side via `getCurrentUser()`, redirects non-admins to `/entry` as a defense-in-depth fallback behind middleware), `app/(admin)/AdminShell.tsx` (client component: top bar with `Wordmark` on dark, admin `RoleLocationBadge`, staff name, log-out button; bottom tab bar — Dashboard/Items/Ingredients/Delivery/Staff — that becomes a top-aligned horizontal nav at `768px+`), `app/(admin)/AdminShell.module.css` (tokens only).
- **`app/api/auth/logout/route.ts`** — new; Phase 2 built `lib/auth.ts`'s `signOut()` but no route ever called it. Needed for the shell's log-out button.
- **`lib/auth.ts`** — added `requireAdmin()`: returns the current user's row if `role === 'admin'`, else `null`. Every admin API route in this phase calls it and returns `403` if it's `null` — the server-side check CLAUDE.md requires on top of RLS, not instead of it.
- **`lib/validation.ts`** — added `itemSchema`, `ingredientSchema`, `deliveryLocationSchema`, `staffCreateSchema`, plus a shared `nonNegativeAmount` Zod field (`.nonnegative()`, message "Must be 0 or greater") used by every price/fee field. Also `itemCategorySchema`/`itemSupplyTypeSchema` enums matching `01_DATA_MODEL.md` exactly.
- **Item Master** — `app/api/items/route.ts` (GET list, POST create), `app/api/items/[id]/route.ts` (PATCH edit); `app/(admin)/items/page.tsx` (server component, initial fetch) + `ItemsClient.tsx` (table, Add/Edit modal using `Card`/`Modal`/`Input`/`Button`/`EmptyState`/`Toast`).
- **Ingredient catalog** — same pattern: `app/api/ingredients/route.ts`, `app/api/ingredients/[id]/route.ts`, `app/(admin)/ingredients/page.tsx` + `IngredientsClient.tsx`.
- **Delivery Locations** — same pattern: `app/api/delivery-locations/route.ts`, `app/api/delivery-locations/[id]/route.ts`, `app/(admin)/delivery-locations/page.tsx` + `DeliveryLocationsClient.tsx`.
- **Staff Management** — `app/api/staff/route.ts` (GET list, POST create-only, no PATCH/DELETE per phase scope). Create follows `scripts/seed-staff.ts`'s exact pattern: service-role client → `nextStaffCode()` (from `lib/staffCode.ts`) → `staffCodeToSyntheticEmail()` → `supabase.auth.admin.createUser()` → insert `public.users` row; if the `public.users` insert fails, the route rolls back the orphaned `auth.users` row via `admin.deleteUser()` so a failed creation never leaves a dangling auth account. Route-level business rules not expressible in the Zod schema (admin ⇒ no location; staff ⇒ location required; `is_store_manager` ⇒ restaurant only, per `00_ARCHITECTURE.md` §5.1) are checked explicitly with clear error messages. `app/(admin)/staff/page.tsx` + `StaffClient.tsx` — read-only list + "Add staff" modal, no edit/delete/PIN-reset UI, matching the phase's explicit exclusion.
- **Dashboard placeholder** — `app/(admin)/dashboard/page.tsx`, an `EmptyState` ("Dashboard coming soon") so admin login has a landing page; the real dashboard is Phase 7. Confirmed with the user before building (see Deviations).
- **Shared admin catalog styling** — `app/(admin)/catalog.module.css` (page header, table per §4.11's comfortable density, form fields, active/inactive badges, error text) — one shared module across Items/Ingredients/Delivery Locations/Staff rather than four near-identical files, all tokens, zero hardcoded values.

**Challenges faced:**
- **No admin (or staff) shell/nav existed anywhere in the repo before this phase** — Phase 2 built only the login screen. This meant the nav shell itself (top bar + bottom tabs) had to be originated from `01_COMPONENTS.md` §4.12's spec directly, with no prior screen to pattern-match against, rather than extending existing admin UI. Handled by building it as its own reusable piece (`AdminShell.tsx`) so Phase 4+ has a template to point to when it needs the equivalent staff-facing shell (bottom nav: Entry/Expenses/Summary, per `CLAUDE.md`'s Project Structure notes).
- **`/entry` doesn't exist yet** (Phase 4 scope) — both `middleware.ts` (Phase 2) and this phase's `app/(admin)/layout.tsx` redirect non-admins there. Confirmed via live testing that the redirect itself fires correctly even though the destination is a 404 right now; this is expected, not a bug — same category of "expected gap" Phase 2's context file already called out for `/entry`/`/dashboard` before this phase existed.
- **Staff creation's rollback path** (deleting the orphaned `auth.users` row if the `public.users` insert fails) isn't something `scripts/seed-staff.ts` needed, since that script just logs and continues on a per-person failure in a batch. The route handler needed it because a single failed request must not silently leave an unusable half-created account behind for the admin to discover later. Not explicitly required by the phase spec, but a direct consequence of "staff account creation" needing to be atomic-in-effect even though it's two separate API calls (Auth admin + Postgres insert) with no shared transaction.
