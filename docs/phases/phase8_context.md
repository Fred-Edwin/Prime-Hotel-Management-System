**Challenges faced:**
- **A real, non-obvious data-model gotcha caused a stuck recording (Video 2, restaurant staff):** seed data for "today" was written by setting the `opening_stock` column directly on a `stock_entries` row. That column is never read back by the actual save path — `save_stock_entry()` (in `supabase/migrations/20260712091633_stock_entry_row_locking.sql`) always recomputes opening stock itself from the **prior day's** `closing_stock`, defaulting to 0 if no prior-day row exists. Since no "yesterday" row existed for the seeded items, real available stock was far smaller than the seed data implied, and the recording's Save click correctly (and confusingly, from the outside) got rejected with a legitimate `409 oversell` error. Root-caused by reading the actual SQL function rather than continuing to guess from the API layer; fixed by inserting a real prior-day row with a proper `closing_stock` for each affected item, rather than ever writing directly into `opening_stock` again.
- **A related process mistake, corrected mid-session:** while diagnosing the above, several throwaway debug scripts made **real, live POST/Save writes** against the production API as a diagnostic technique, which made the problem worse (ate further into the small real stock ceiling) rather than just observing it. Corrected by switching to read-only GETs (via an authenticated session cookie) for all further diagnosis, and explicitly avoiding any further ad-hoc writes against production as a debugging tool.
- **Vercel's `vercel env pull --environment=production` returned empty-string values** for `NEXT_PUBLIC_SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` despite `vercel env ls` showing them present and correctly "Encrypted" — cause not fully root-caused (possibly a CLI-side cache/bug), worked around by using the real production credentials that were already present in `.env.local` (lines 6-13, a second block below the local-Docker block on lines 1-3) — confirmed genuinely live via a direct REST call returning `200`.
- **Checkpoint-screenshot handling changed direction mid-phase:** checkpoints were initially treated as disposable verification artifacts and deleted after each recording; the user corrected this ("keep the checkpoint images"), so all 4 recordings' checkpoints were kept through completion and used for verification; then, once all videos were confirmed correct, the user asked for them to be removed after all — they were deleted as the very last step of this phase. Net effect: only the Admin video ever had its checkpoints deleted "early" (before the keep-instruction existed); the other 3 had checkpoints kept through the full verification cycle and removed only at final cleanup, which is the intended sequence.
