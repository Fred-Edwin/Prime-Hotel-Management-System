*Item 1 — batch-save performance fix (root cause + fix authored in a prior same-day session; this session did the actual verification, which had been blocked):*
- `supabase/migrations/20260713183705_batch_save_functions.sql` — three plpgsql wrapper functions (`save_stock_entries_batch()`, `save_canteen_stock_entries_batch()`, `save_ingredient_entries_batch()`) that accept a whole batch as `jsonb` and loop **server-side**, calling the existing single-row `save_stock_entry()`/`save_canteen_stock_entry()`/`save_ingredient_entry()` per line inside one transaction. - **Local dev auth root-caused and fixed for real** (the actual blocker from the prior session's handoff): `.env.local` had two blocks both defining `SUPABASE_SERVICE_ROLE_KEY` (local Docker keys on lines 1–3, production keys on lines 5–7 kept since Phase 8 because `vercel env pull` was broken). `dotenv` keeps the **last** value on a duplicate key, so every local script — including `seed-staff.ts` and, critically, the app's own route handlers reading `process.env.SUPABASE_SERVICE_ROLE_KEY` — was silently getting the **production** service-role key at runtime, not the local one. This is what broke local auth: a production-signed JWT rejected by the local GoTrue instance. The prior session's diagnosis (an EC/HS256 signing-key mismatch in GoTrue) was a red herring — direct REST calls to the local auth admin API worked fine with the correct local key. **Fixed by renaming the production block's keys** to `SUPABASE_ANON_KEY_PROD` / `SUPABASE_SERVICE_ROLE_KEY_PROD` in `.env.local`, confirmed nothing in the codebase referenced the un-suffixed prod names (only manual/ad hoc prod scripts ever used them).

Both resets applied all 19 migrations cleanly including both of this phase's new ones.
- `lib/supabase/types.ts` regenerated (via `npx supabase gen types typescript --db-url ...` — the `--local` flag failed with a stale "not running" error despite the stack being up; `--db-url` against `127.0.0.1:54322` worked) to add the three new RPC signatures plus, later in the session, `users.active`. *Item 3 — staff account management (create-only → full CRUD, new this session):*
- `supabase/migrations/20260713221946_users_active_soft_deactivate.sql` — adds `users.active boolean not null default true`. Soft-deactivate chosen over hard-delete because `stock_entries.created_by`/`ingredient_entries.created_by`/`expenses.created_by`/`orders.created_by` have no `ON DELETE CASCADE`/`SET NULL` — confirmed by reading the schema, not assumed.
- `app/api/auth/login/route.ts` — now looks up `active` alongside `staff_code` and rejects a deactivated account's login attempt with the same generic `401 "Name or PIN is incorrect"` as a wrong PIN (deliberately not a distinguishing message, so a deactivated staffer can't tell "wrong PIN" from "account disabled" by probing).
- `lib/validation.ts` — new `staffUpdateSchema` (name/role/location/store-manager/active) and `staffPinResetSchema` (pin only, kept separate from the edit schema so an unrelated field edit can never accidentally touch the PIN).
- `app/api/staff/[id]/route.ts` (new) — `PATCH`, admin-only, re-validates the same cross-field business rules as staff creation (admin has no location, staff must have one, only restaurant staff can be store manager) since a role/location change can invalidate a previously-valid combination. Blocks an admin from deactivating their own account (`400`).
- `app/api/staff/[id]/pin/route.ts` (new) — `POST`, admin-only, calls `serviceClient.auth.admin.updateUserById()` to reset the password/PIN — a distinct route from the field-edit `PATCH` since it needs the service-role Auth admin API, not a plain table update.
- `app/(admin)/staff/StaffClient.tsx` — added a Status column (Active/Deactivated badge, reusing `catalog.module.css`'s existing `badgeActive`/`badgeInactive` classes), Edit and Reset PIN row actions, and two new modals (edit fields including an Active checkbox; PIN reset).
- `app/api/staff/route.ts` and `app/(admin)/staff/page.tsx` — `active` added to the `GET`/initial-load column selections.
- **Real discovery made while testing, not anticipated in planning:** Supabase Auth's `minimum_password_length=6` (`supabase/config.toml`, mirrored in production) is enforced by `admin.updateUserById()` (PIN reset) but **not** by `admin.createUser()` (staff creation, and `scripts/seed-staff.ts`) — an asymmetry between the two Auth admin API calls. First PIN-reset attempt with a 4-digit PIN threw an unhandled `AuthWeakPasswordError`, surfaced to the admin as a raw `500`. **Fixed by tightening `staffCreateSchema`/`staffPinResetSchema` to require exactly 6 digits** (matching the real, already-production-enforced constraint — Phase 8 had already reset all real production PINs to 6 characters for this same reason), so both routes now fail fast client-side with a clear message instead of a confusing 500 three steps later. `loginSchema` (validating login *input shape*, not creating a credential) deliberately stayed at the looser 4–6 digit range, since it must still accept any already-existing PIN, including dev seed data's legacy 4-digit convention.
- `scripts/acceptance/phase9-staff-orders.mjs` (new) — 20 checks covering both items 2 and 3 together: admin order list/filter/403-for-staff, staff creation PIN-length validation, edit, PIN reset (old PIN stops working, new PIN works), the mandatory deactivation-blocks-login check (with reactivation confirmed to restore access), and admin-cannot-self-deactivate.
- Verified via `curl` for every code path (edit, PIN reset, deactivate/reactivate, self-deactivation block) and visually via the `verify` skill (staff list with Status column, edit modal pre-populated correctly).

**Challenges faced:**
- **The local dev auth root-cause misdiagnosis** (see Deviations #3) cost real time before the actual fix was found — a lesson in verifying a claimed root cause directly (a raw `curl` against the Auth admin API) rather than proceeding on a prior session's diagnosis, even when it's plausible and specific.
- **A self-inflicted test-data drift mid-session:** curl-testing the new PIN-reset feature on the real seeded "Mercy Wanjohi" account (rather than a disposable fixture) left her PIN inconsistent with `scripts/acceptance/_lib.mjs`'s hardcoded roster PINs, breaking `phase6-orders.mjs`'s login step. - **`npx supabase gen types typescript --local` failed with a stale "not running" error** despite `npx supabase status` showing all containers healthy; switching to `--db-url postgresql://postgres:postgres@127.0.0.1:54322/postgres` worked immediately. Not root-caused further (a CLI-version quirk, not a project issue) — noted here so a future session doesn't re-diagnose it from scratch.
- **The types-regeneration file-overwrite near-miss:** the first `gen types --local` attempt wrote its own JSON error object over `lib/supabase/types.ts` (redirected stdout unconditionally), destroying the real file's content. Caught immediately via `git diff --stat` showing a 965-line deletion; restored with `git checkout -- lib/supabase/types.ts` before any further action. Every subsequent regeneration was written to a scratch path first and diffed before ever touching the real file.
