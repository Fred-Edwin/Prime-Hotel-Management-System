**Deviations from the phase plan / judgment calls made (flagged per the brief):**

1. **No stored `orders.payment_status` column** — outstanding balance is derived entirely at query/read time (`total_amount - sum(order_payments.amount)`), never denormalized. The brief explicitly allowed either choice provided a stored field is kept correctly in sync via a recompute function; I judged the derive-only approach sufficient at this business's actual scale (low hundreds of entries/day) and avoided building a second historical-recompute mechanism for a marginal read-performance win. Documented with full reasoning in the migration's file header and in `01_DATA_MODEL.md` §6.

2. **`order_payments` has no client-facing INSERT RLS policy at all** — every other append-only table in this schema (`ingredient_purchases`, `canteen_stock_purchases`) exposes a plain RLS-gated INSERT alongside its write function. `order_payments` doesn't, because recording a payment *must* pass through the overpayment recheck + advisory lock inside one transaction (`record_order_payment()`), and a bare client-side insert has no way to enforce "don't exceed the outstanding balance" itself. This is a deliberate asymmetry from the existing pattern, documented inline in the migration and in `01_DATA_MODEL.md` §4.

3. **A `'counter'` order behaves identically to `'pickup'` for stock-deduction and fee purposes** — I did not add any new branching to `apply_order_to_stock_entry()` or `create_order()`'s core logic; `'counter'` simply falls through the existing `fulfillment_type !== "delivery"` paths in `app/api/orders/route.ts` and `create_order()`'s SQL (no delivery zone/fee). This was implied but not explicitly spelled out in the brief — confirmed correct by re-reading `00_ARCHITECTURE.md` §13/`01_DATA_MODEL.md` §6's existing "walk-in till sales are unaffected, orders are a second write-path" framing, which treats every fulfillment type identically for the stock mechanism.

4. **"On credit" is a UI-only concept, not a stored flag.** There is no `orders.is_credit` column — the only durable signal that an order might be a credit sale is whether `customer_id` is set (a cash order can *also* carry a `customer_id`, per the brief's explicit allowance — "linking a cash order to a customer is still allowed if useful"). The staff-facing order list's "Customer on file" badge and the whole debtors screen both key off `customer_id` + the derived outstanding balance, never a stored intent flag. This follows directly from decision #4 in the brief's "already agreed" list but is worth flagging as the concrete implication: there is no way to distinguish, after the fact, "this cash order happened to be linked to a customer" from "this was logged with credit intent but got paid off immediately" — both end up as `customer_id` set + `outstanding = 0`. Judged acceptable since the debtors screen only ever surfaces orders with `outstanding > 0` anyway, so a fully-paid order's original intent is moot for the feature's actual purpose.

5. **Debtors nav item reuses the existing `expenses` icon** (a wallet/receipt shape) rather than a new dedicated icon — `components/Icon/Icon.tsx` has no debt/credit-specific icon and the brief didn't ask for new iconography. Flagged as a minor design-system gap per CLAUDE.md's "flag it explicitly, don't silently invent a new pattern" rule, not silently worked around with a new SVG.

6. **"Debtors" was NOT added to `AdminShell`'s `PRIMARY_MOBILE_NAV_HREFS`** (the four bottom-tab slots on mobile) — it lives under "More" alongside Ingredients/Purchases/Delivery/Audit Log. This is a reasonable default (Dashboard/Ledger/Orders/Staff were human-confirmed as the primary four per `AdminShell.tsx`'s own comment, and I didn't re-litigate that confirmed list unprompted), but if WaPrecious checks debtors frequently in practice this may deserve promotion — noted for a human decision, not decided silently.

7. **`customer_id` existence check in `POST /api/orders`** queries `customers` without a location filter (any staff member can link any customer, matching the "customers aren't location-scoped" design decision) — an intentional widening beyond how most other FK-reference checks in this codebase work, consistent with §6's explicit "a customer isn't necessarily tied to one location, don't force it" instruction.

**Challenges faced:**
- **Postgres enum-value-in-same-transaction constraint** — same issue the codebase had already solved once (`20260713120000_add_item_categories.sql`). Found the precedent early by grepping existing migrations before writing anything, so this wasn't rediscovered the hard way.
- **Choosing the right SQLSTATE codes without colliding with existing ones** — `P0004` was already claimed by `create_staff_meal_entry()`'s `unknown_item` case; used `P0005`/`P0006` instead and documented the collision-avoidance reasoning inline in both the migration and `lib/errors.ts`, per this codebase's existing "distinct SQLSTATE per rejection reason" convention.
- **The lint rule for setState-inside-effect** (`react-hooks/set-state-in-effect`) initially failed on `DebtorsClient.tsx`'s `useCallback`-based loader pattern — fixed by switching to the same nested-function-inside-the-effect shape `OrdersClient.tsx` (Phase 6) already uses, which the linter doesn't flag. Caught by running `pnpm lint` before considering the screen done, not assumed clean.
- **Session interruption/resume** — this session was restarted mid-task by the harness (not a self-inflicted issue). Recovered by checking `git status`/`git diff --stat` and re-reading the two migration files already on disk before continuing, confirming nothing already written had been lost, rather than restarting from scratch or risking duplicate/conflicting work.
- **A real bug found during post-migration curl verification: `record_order_payment()` was initially shipped as `security invoker`, and every legitimate payment attempt failed with a 403.** Root cause: `order_payments` deliberately has no INSERT policy at all (so a bare client insert is always rejected, forcing every write through the function) — but `security invoker` means the function runs *as the calling user*, so its own `insert into order_payments` is subject to that exact same "no INSERT policy" RLS rejection. The overpayment check (which runs *before* the insert) worked correctly and was never the problem; only the insert itself was blocked. Root-caused by comparing the actual Postgres error surfaced through `describeSaveError()`'s generic `42501` branch ("You can only save today's entry") against what the code path should have raised (`P0005`/`P0006`) — the mismatch pointed straight at an RLS rejection happening somewhere `record_order_payment()`'s own explicit checks don't raise. Fixed by changing the function to `security definer` (+ `set search_path = public`, the required safety pairing) so it runs with elevated privilege and can perform the insert despite the table's closed RLS — with the function's own explicit `is_admin() or location = my_location()` check remaining the real authorization boundary, same shape as this schema's existing `security definer` precedent (`write_audit_log()`, `canteen_supplied_total()`, `01_DATA_MODEL.md` §4) rather than `record_ingredient_purchase()`'s `security invoker` precedent, which doesn't apply here since that case relied on an UPDATE policy that was correctly widened to permit the caller directly — there's no equivalent "a direct client insert should be allowed" case for `order_payments`. The human had already applied the original (buggy) migration to `prosper-hotel-dev`; rather than re-running the whole file, only the corrected `create or replace function public.record_order_payment(...)` block was handed back for the human to re-apply (idempotent, safe against the already-existing table/policies). Re-verified via curl immediately after — partial payment succeeded (201) on the first attempt post-fix. `docs/01_DATA_MODEL.md` and the migration file's own comments were both corrected in place to describe the fixed (`security definer`) version as the actual, current behavior — not left describing the broken version that was briefly live.
- **The new acceptance script's first two runs surfaced two more real issues, both in the *test*, not the feature** — worth recording since they're easy to mis-diagnose as feature bugs on a future re-run: (1) comparing a shared seed item's `quantity_sold` across two separate GET calls a few hundred ms apart produced a stale/out-of-order read on the second run (any other order/till-sale touching that same item concurrently makes this comparison inherently racy) — fixed by giving the script its own dedicated fixture item instead of reading a real seed item's cumulative figure; (2) a brand-new fixture item has no `stock_entries` row yet, so ordering against it immediately hit a genuine, correct oversell rejection (409) — fixed by seeding a `stock_entries` row via the service-role client first, same pattern `post-launch-staff-meals.mjs` already established for exactly this reason.
