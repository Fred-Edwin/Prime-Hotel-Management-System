**What was built:**
- **`supabase/migrations/20260712080310_orders_write_function.sql`** — two new Postgres functions:
  - `public.apply_order_to_stock_entry()` — the "second writer" half of §3.4's mechanism. Called once per distinct item on a newly created order. Does the same opening-stock-carry-forward-and-upsert work `save_stock_entry()`/`save_canteen_stock_entry()` already do, but never writes `till_quantity_sold`/`sent_out`/`wastage` — only `quantity_sold` (and, for `canteen_supplied` items, `added_stock`) and their downstream values move, always re-derived from a fresh sum of `order_items`. Idempotency via a caught `unique_violation` on `orders(created_by, client_request_id)`: a retried submit returns the already-created order instead of erroring or duplicating.

**Deviations from the phase plan / data model:**
1. **`public.apply_order_to_stock_entry()` was built instead of calling the originally-planned `public.recalculate_stock_entry()` directly**, and `docs/01_DATA_MODEL.md` §3.4 has been updated in this same phase to document why (a genuine, documented architecture change, not a silent divergence — per CLAUDE.md's rules for handling data-model changes mid-project). `recalculate_stock_entry()` is `UPDATE`-only and assumes a `stock_entries` row already exists; an order can be the first write of the period for an item (e.g. a delivery placed before the till sheet is touched that day), which the stub can't handle — there's no row to `UPDATE` and no column defaults for `closing_stock`/`sales_value`/etc. to fall back on. `apply_order_to_stock_entry()` does the same full upsert `save_stock_entry()` does, restricted to never touching the fields the till-entry flow owns. `recalculate_stock_entry()` itself is left in the schema, unused — harmless, not deleted, since deleting it wasn't this phase's job and it's not causing any drift.
2. **`apply_order_to_stock_entry()` needed to be cadence-aware for canteen orders**, which wasn't explicitly anticipated in the original §3.4 design (written before Phase 5's weekly-cadence work existed). This was found as a genuine bug during this phase's own live testing (see "Challenges faced"), not designed in from the start — fixed within this phase, also documented in §3.4's implementation note.
3. **`GET /api/orders` also returns today's/this-week's `stock_entries` rows**, not just items/zones/orders as originally scoped — added specifically so the order screen's item steppers could show and enforce a live "Available: N" cap, matching the design-system checklist's "oversell visually prevented before it's attempted" requirement that Entry's screen already meets. Not a data-model change (no new table/column), just a slightly larger read in an existing route.
4. **Two shared-component fixes** (`TillStrip`'s `disabled` prop, `Input`/`Select`'s `width: 100%; min-width: 0`) — both are corrections of latent gaps in Phase 1's component library, surfaced by this phase being the first to (a) need a "can't save yet, not currently saving" distinction, and (b) place `Input`/`Select` inside a constrained-width grid row. Neither changes any existing screen's rendered output (confirmed via the full test suite + no visual regression in Entry/Store/Expenses' existing screenshots' underlying CSS — `disabled` defaults false, `width:100%` is a no-op in every previously-full-width usage).

### The two mandatory tests, in full

Both were written as a real, repeatable Node script (`orders-tests.mjs`, run via `node` against the live dev server + local Supabase stack, logging in as real seeded roster accounts over HTTP — not browser-driven, since these are API-level correctness checks, not UI checks) rather than "tried it once and it looked fine." The script isn't checked into the repo (throwaway test harness for this session, per the instructions' framing of it as a one-off acceptance check rather than a permanent CI test) — reproducible from this description if needed again.

**Test 1 — Concurrency:** fired a till "Save entry" (`till_quantity_sold: 3` for one item) and a delivery order (`quantity: 2`, same item/location/day) via `Promise.all` so they hit the server essentially simultaneously. Result: `quantity_sold: 5` (3+2, both writes present), `till_quantity_sold: 3` exactly preserved (the order never touched it). Confirms neither write-path clobbered the other's contribution.

**Test 2 — Duplicate submission:** submitted one order, then retried the identical payload (same `client_request_id`) twice more. All three responses returned the same `order.id`. `stock_entries.quantity_sold` incremented by exactly 1 (the order's quantity), not 3 — confirming the retries were true no-ops at the stock-deduction level, not just at the "didn't create a second orders row" level.

**Additional scenarios tested per the brief's "what's the obvious but untested case here" prompt:**
- Two genuinely separate orders from the same customer, same day, different `client_request_id`s: both succeeded with **different** order ids — confirmed the idempotency key doesn't accidentally dedupe legitimate repeat business.
- Zero-item order: rejected with 400 (Zod's `.min(1)`).
- `canteen_independent` item (Bananas) submitted on a restaurant order: rejected with 400 and a specific message, not a generic error.
- An order as the literal first write of the day for an item (no prior till entry): succeeded, correctly derived `opening_stock` from the prior day's `closing_stock` (carry-forward), created the `stock_entries` row itself with correct price snapshots and `closing_stock` — this is the exact scenario that proved `recalculate_stock_entry()` alone was insufficient (see "Deviations" #1).
- A canteen order for a `canteen_independent` item: correctly folded into the existing weekly `stock_entries` row (Monday `entry_date`), not a new stray daily row — this is the scenario that surfaced the cadence bug described in "Challenges faced" and "Deviations" #2.
- A canteen order for a `canteen_supplied` item: correctly re-derived `added_stock` via `canteen_supplied_total()` on the order write path, matching what a weekly till save would independently compute (verified both return the same value from a shared fixture).
- Cross-location RLS: canteen staff's `GET /api/orders` list does not include a restaurant order placed moments earlier, confirmed at both the route and RLS-impersonated table level.

**Challenges faced:**
- **The canteen cadence bug** (Deviations #2) — `apply_order_to_stock_entry()`'s first version used the order's own `order_date` as the `stock_entries.entry_date` unconditionally, correct for restaurant (daily) but wrong for canteen (weekly). A canteen order created a stray extra `stock_entries` row at the literal order date instead of folding into the existing week-Monday row, so the item's stock effect became double-represented across two rows — a real data-integrity bug, not a cosmetic one. Found by deliberately testing a canteen order end-to-end (not assumed safe by analogy to the restaurant case) and cross-checking the raw `stock_entries` table via direct SQL, which showed two rows where there should have been one. Fixed by making the function resolve to `date_trunc('week', order_date)::date` for canteen, matching the same convention the existing same-week RLS policies (`20260711150001`/`20260711160001`) and `save_canteen_stock_entry()` already use, and re-deriving `added_stock` via `canteen_supplied_total()` the same way. This is exactly the class of bug Phase 5's context file's "Instructions for the next phase" flagged as a risk ("orders don't need their own weekly convention... just flagging Phase 6 shouldn't assume") — worth noting that flag correctly identified that `orders.order_date` itself doesn't need special handling, but didn't anticipate that the *stock_entries write triggered by an order* would need the weekly resolution applied at the point of writing, which is what actually broke.
- **The Fee-field grid overflow** (Deviations #4) — per CLAUDE.md's "verify, don't guess" rule, a screenshot at the standard 390px staff viewport during self-review (not prompted by a bug report — this was found proactively during the checklist pass, not after a failed first attempt) showed the delivery zone's Fee input visibly clipped off the right edge of the card. Root cause via `boundingBox()` inspection (not guessed from source): `Input`'s `.field` div had no explicit width, so inside the two-column CSS grid its child `<input>`'s intrinsic sizing pushed it past its grid track (a flex item's default `min-width: auto` lets content override an explicit narrower container). Fixed at the component level (`width: 100%; min-width: 0`) rather than working around it locally in `orders.module.css`, since the same latent bug would recur the first time any future screen placed `Input`/`Select` in a constrained-width row — re-verified with a second screenshot and a `boundingBox()` check showing both grid children now correctly sized at ~152px each within the 316px row.
- **A misleading "Saving…" label on a disabled-but-idle save button** — also found during the self-review screenshot pass, not reported by testing. `TillStrip`'s only disable mechanism (`saving={true}`) also swapped the label text, so an order with no customer name/items yet showed "Saving…" on a button that wasn't saving anything. Fixed by adding a distinct `disabled` prop to the shared component rather than working around it in `OrdersClient` (e.g. by passing a fake "saving" state), since conflating "can't submit yet" with "currently submitting" is a real, reusable UX defect any future screen using `TillStrip` for a validated (not just always-ready) form would hit again.
- **Pre-existing dev-seed data caused one false-alarm test failure** — an early concurrency-test run reported the till-save half failing with a 403 ("You can only save today's entry"), which first looked like a regression. Root cause (found via direct SQL, not assumed): one `stock_entries` row for that day already existed from an earlier phase's live-testing session, created by a *different* staff account (Janiffer, not the test's Sarah), and the batch-save route's per-line loop aborted the whole request on that one line's ownership-check failure — a pre-existing multi-writer-within-the-same-batch limitation of `save_stock_entry()`'s design (not part of §3.4's till-vs-orders mechanism, and not introduced by this phase), surfaced only because of leftover local dev data. Cleared the stale test data and re-ran clean; not a real Phase 6 bug, and no code change was made for it (documented here so a future session doesn't waste time re-discovering it against the same stale fixture, if it recurs).

**Post-phase follow-up (2026-07-12, same day, different session — logged as a diary entry per CLAUDE.md's "never edit a previous phase's context file" rule, not folded into the sections above):**

While rebuilding this phase's lost acceptance-testing coverage as a real, repeatable script (`scripts/acceptance/phase6-orders.mjs` — see `scripts/acceptance/README.md` for why these now exist: Phases 4–6's original RLS/concurrency verification was all one-off `curl`/Node sessions never saved past the session that wrote them), a **real concurrency bug was found and fixed**, not just re-verified:

- **The bug:** `save_stock_entry()`, `save_canteen_stock_entry()`, and `apply_order_to_stock_entry()` each checked "does a `stock_entries` row already exist" via a plain, non-locking `SELECT`, computed their oversell check from that snapshot, then `INSERT ... ON CONFLICT DO UPDATE`. When two calls raced as the **simultaneous first-ever write** for a brand-new row (a till save and a delivery order both landing before either had ever touched that item/day), Postgres serializes the actual row conflict, but does not re-run the PL/pgSQL function body for the blocked caller — its `ON CONFLICT` `EXCLUDED` values were already computed from stale pre-block data. Result: a legitimate order could be wrongly rejected with a false `409` oversell error.
- **Why the original Phase 6 testing missed it:** last session's ad hoc concurrency test happened to run against an item that already had a `stock_entries` row from earlier test data (`opening_stock: 15` already committed), so the race was between two `UPDATE`s on an existing row — safe by construction. This session's acceptance script ran against a genuinely fresh local database with no leftover rows, which is what exposed the real first-write-vs-first-write race.
- **The fix:** `supabase/migrations/20260712091633_stock_entry_row_locking.sql` — a new `public.lock_stock_entry_row(item_id, location, entry_date)` helper taking a `pg_advisory_xact_lock` keyed on that triple, called as the very first statement in all three write functions, before any read. This serializes each function's entire read-decide-write sequence per row, not just the final `INSERT`'s conflict resolution. Documented in full in `docs/01_DATA_MODEL.md` §3.4's new "Row-locking fix" note (updated in the same session per CLAUDE.md's mid-project architecture-change rules).
- **Verification:** the original repro (concurrent till-save + order on a brand-new row) re-run 8+ times back to back post-fix, zero false rejections, correct combined `quantity_sold` every time. `scripts/acceptance/phase6-orders.mjs`'s full 24-check suite passes twice in a row with full self-cleanup. Full static suite (`pnpm typecheck`/`lint`/`test`/`build`) re-run clean after the migration.
- **Not yet done:** this fix hasn't been pushed to a deployed environment (this was a local-only session) — the next session that pushes to `main` should confirm CI is green on this migration same as any other, and if a staging/production Supabase project already exists by then, this migration needs to be applied there too, not just locally.
