# Phase 4 ‚Äî Restaurant Daily Entry ‚Äî 2026-07-11

**Status:** Complete

**What was built:**
- **`lib/calculations.ts`** ‚Äî the single source of truth for ¬ß3's formulas: `totalStock`, `calculateStockEntryTotals` (closing_stock, sales_value, cost_value, closing_stock_value, wastage_value), `isStockEntryOversold`, and the ingredient-side equivalents `calculateIngredientEntryTotals`/`isIngredientEntryOversold`. Covered by `lib/calculations.test.ts` (8 tests), including an explicit test that wastage is valued at buying price, never selling price.
- **New migration `supabase/migrations/20260711100001_entry_write_functions.sql`** ‚Äî two new Postgres functions, `public.save_stock_entry(...)` and `public.save_ingredient_entry(...)`. Each does the *entire* write atomically in one round trip: derives `opening_stock` from the prior period's `closing_stock` (¬ß3.1), computes `quantity_sold` as `till_quantity_sold + sum(order_items)` (¬ß3.4, same join `recalculate_stock_entry()` uses), re-validates the combined oversell check post-computation, and upserts the row (`on conflict (item_id, location, entry_date)` / `on conflict (ingredient_id, entry_date)`) ‚Äî all `security invoker`, so existing RLS policies still gate who can call it. Regenerated `lib/supabase/types.ts` after applying.
- **`lib/validation.ts`** additions ‚Äî `stockEntryLineSchema`/`stockEntriesSaveSchema`, `ingredientEntryLineSchema`/`ingredientEntriesSaveSchema`, plus a shared `nonNegativeQuantity` field (parallel to Phase 3's `nonNegativeAmount`).
- **`app/api/stock-entries/route.ts`** ‚Äî `GET` (items scoped to the caller's location's sellable supply_types + any existing entries for the date) and `POST` (batch save, one `save_stock_entry` RPC call per line, snapshotting `items.selling_price`/`buying_price` at write time). Oversell errors from the RPC surface as `409` with a plain-language message, not the raw Postgres exception.
- **`app/api/ingredient-entries/route.ts`** ‚Äî same GET/POST shape, gated to `role === 'staff' && location === 'restaurant' && is_store_manager` in the route handler itself (RLS already restricts `ingredient_entries` reads/writes to restaurant, but the store-manager narrowing is a UI/route rule per `00_ARCHITECTURE.md` ¬ß5.1, same pattern as Phase 3's admin routes).
- **`app/(staff)/StaffShell.tsx` + `.module.css`** ‚Äî staff-facing nav shell, following `AdminShell.tsx`'s exact structural pattern (top bar + bottom tabs, `768px+` reflow to a horizontal top nav) per Phase 3's explicit instruction. Nav: Entry / Expenses / Summary, plus Store appended only when `is_store_manager` is true. Light (`color-surface-raised`) top bar per ¬ß4.12, vs. `AdminShell`'s dark aubergine bar ‚Äî staff and admin are visually distinct surfaces.
- **`app/(staff)/layout.tsx`** ‚Äî server component, re-checks `role === 'staff' && location` server-side (same belt-and-suspenders pattern as `app/(admin)/layout.tsx`), wraps children in `StaffShell`.
- **`middleware.ts`** ‚Äî added the missing reverse gate: admin hitting a staff route prefix (`/entry`, `/store`, `/expenses`, `/orders`, `/summary`) now redirects to `/dashboard`, symmetric with the existing staff-off-admin-routes rule. This gap existed since Phase 2/3 (only one direction was enforced) ‚Äî closed here since it's now directly testable with real staff routes in place.
- **`app/(staff)/entry/page.tsx` + `EntryClient.tsx` + `entry.module.css`** ‚Äî the daily restaurant entry screen. Category-grouped via `CategoryChips` (all/beverages/snacks/meals/...), one row per item showing name, selling price, read-only "Opening: N" line, a `LowStockIndicator` dot when remaining stock is at or below a 5-unit threshold. Non-store-manager staff see a single till-quantity `Stepper` per row (their primary action); the store-manager-flagged user instead sees **Added stock** and **Sent to canteen** steppers as the visually primary fields, with **Quantity sold** demoted to a secondary (slightly deemphasized) field below ‚Äî per the phase spec's "store-manager-emphasis variant." Wastage is behind a de-emphasized "Log wastage" toggle per row (stepper + optional note `Input`), collapsed by default unless already non-zero. A sticky `TillStrip` at the bottom shows live item count + running sales value and does one batch "Save" rather than per-tap requests.
- **`app/(staff)/store/page.tsx` + `StoreClient.tsx`** ‚Äî the store manager's ingredient entry screen, reusing `entry.module.css` (identical light-surface visual treatment, same "Daily entry" pattern per `docs/SCREENS.md`). Server-redirects any non-store-manager to `/entry`. Rows show ingredient name, unit, opening stock; primary fields are **Received** and **Used in cooking**; wastage is the same collapsed toggle pattern. Till strip shows item count + total ingredient cost used (not a sales value, since ingredients have no selling price).

**UI-expert lens adopted:** POS/retail UX specialist, per `CLAUDE.md`'s "Building a real screen" protocol ‚Äî `/entry` is used multiple times a day under time pressure, so speed and error-prevention (steppers capped at remaining stock, oversell rejected with a specific "only N left" message, batch save instead of per-tap network round trips) took priority over information density.

**Deviations from the phase plan:**
1. **A new migration was added** (`20260711100001_entry_write_functions.sql`, two new Postgres functions) ‚Äî not explicitly listed in the phase plan's scope bullets, which only said "API route... calls `recalculate_stock_entry()`." Reason: PostgREST/the Supabase JS client has no multi-statement client-driven transaction, and ¬ß3.4 requires the upsert + recalculate + oversell re-check to be atomic. Raised to the user as an explicit choice before building (sequential JS calls with a narrow partial-failure risk, vs. one new RPC function) ‚Äî user chose the RPC function. This is a schema addition per `CLAUDE.md`'s mid-project change rules; documented here and in `01_DATA_MODEL.md` shouldn't need edits since it's additive machinery consistent with the already-documented `recalculate_stock_entry()` pattern, not a change to any documented table/column/formula.
2. **`middleware.ts` gained a new admin-off-staff-routes redirect**, not explicitly asked for by the phase plan, but a direct, symmetric consequence of staff routes now existing for real ‚Äî the reverse case (staff off admin routes) was already enforced since Phase 2/3. Verified live (admin hitting `/entry` ‚Üí `307` to `/dashboard`).
3. **`EntryClient` takes no `location` prop** ‚Äî initially planned to pass the staff member's location through for a possible canteen-specific rendering branch, but Phase 4 is restaurant-only (`04_PHASE_PLAN.md` explicitly excludes canteen's entry flow); kept the component to only what Phase 4 actually needs rather than half-building a Phase 5 branch. Phase 5 will decide whether canteen's weekly screen is a variant of this component or its own, per its own scope.

**Challenges faced:**
- **No multi-statement client transaction available** for the ¬ß3.4 atomic-write requirement ‚Äî the main design decision this phase, resolved by adding `save_stock_entry()`/`save_ingredient_entry()` as new `security invoker` Postgres functions rather than sequential JS calls. Chose to append the optional `p_wastage_note` parameter at the end of each function's argument list with `default null`, since Postgres requires defaulted parameters to trail non-defaulted ones ‚Äî a small SQL-signature detail, not a design change.
- **Supabase's generated RPC arg types don't accept `null` for a defaulted text parameter** ‚Äî only `string | undefined`. Fixed by passing `?? undefined` instead of `?? null` at the two call sites (`app/api/stock-entries/route.ts`, `app/api/ingredient-entries/route.ts`) rather than loosening the Zod schema or the SQL function signature.
- **`npx supabase gen types` occasionally intermixed its own stderr progress line ("Connecting to db 5432") into the redirected stdout file** the first time it was run this session, corrupting `lib/supabase/types.ts` and briefly breaking lint with a parse error. Not a real code issue ‚Äî re-ran with stderr redirected separately and confirmed clean output before overwriting the file. Worth remembering for future phases: always sanity-check the first few lines of a freshly generated `types.ts` before relying on it.
