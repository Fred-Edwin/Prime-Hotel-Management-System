## Open design call — NOT yet decided, get the user's answer before implementing

**Oversell diagnosis for canteen: does the restaurant's "ask the store manager" fix even apply here, and if not, what should replace it?**

Context: the restaurant's cashier redesign fixed a real bug where the first writer of the day (a cashier logging a till sale before the store manager logs `added_stock`) got a misleading generic oversell message instead of a message diagnosing the real cause. That fix doesn't map cleanly onto canteen:

- Canteen has **one person** (Anne) filling in both fields, usually in one sitting, once a week — the restaurant's "two different people, different times" race that motivated the original fix is much less likely here in the exact same shape.
- But canteen's `added_stock` for `canteen_supplied` items has its **own** external dependency: it's server-derived from `canteen_supplied_total()`, summing the restaurant's daily `sent_out` for that week. If Anne saves a `quantity_sold` for a `canteen_supplied` item before the restaurant has sent anything that week, the oversell check will fail — not because Anne did anything wrong, but because the restaurant hasn't supplied yet. This is a genuinely analogous "the failure isn't the current user's fault" case, just with a different upstream actor (the restaurant's daily sends, not a store manager's same-screen field).
- For `canteen_independent` items, there's no such external dependency — Anne owns both `added_stock` and `quantity_sold` herself for those, so a real oversell there probably *is* just a real oversell (she's asking to sell more than she said she received), and the generic message is likely already correct and adequate.

**Do not decide this yourself and implement it silently.** Bring this specific question to the user before writing the SQL/route layer: should `canteen_supplied` items get a distinct "the restaurant hasn't sent this week's supply yet" message (mirroring the restaurant's `P0002`/`not_yet_stocked` pattern), while `canteen_independent` items keep the generic oversell message? Or is the generic message acceptable for both, given canteen's single-person-both-fields shape makes the false-diagnosis risk lower overall? State the tradeoff plainly and let them pick — this is exactly the kind of call the restaurant session flagged rather than assumed, and got useful signal back for it.
